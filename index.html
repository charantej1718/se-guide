Introduction to Software Engineering

What is Software Engineering?

Software Engineering is the application of systematic, disciplined, and quantifiable approaches to the development, operation, and maintenance of software. It bridges the gap between computer science theory and real-world software building.

Course Overview — What You Will Learn

📘 SDLC

Software Development Life Cycle — the entire process from idea to delivery

📐 UML

Unified Modelling Language — diagrams for design documentation

🏗️ Architecture

How large systems are organized and structured

🧪 Quality

Testing, metrics, and management for high-quality software

Course Outcomes (COs)

CO2Apply SDLC process to a software project
CO3Apply SE tools and techniques for requirements analysis, design, and documentation
CO4Develop high-quality software within time and budget estimates
CO5Monitor software quality at various stages of SDLC

Process Models

Software Process Phases

PhaseActivities
📋 Definition (Specification)Requirements analysis, project planning, system analysis
🔨 DevelopmentSoftware design (high-level + detailed), coding, unit testing, integration
✅ ValidationSystem Integration Testing (SIT), User Acceptance Testing (UAT), QA Testing
🔄 Evolution (Maintenance)Correction, adaptation, enhancements, prevention

Process Models Compared

ModelKey IdeaBest ForWeakness
🌊 Waterfall Sequential — each phase must complete before next begins. Each phase produces a "signed-off" document. Embedded systems, critical systems where requirements are stable Cannot go back to modify; inflexible to changing requirements
🔁 Incremental Develop software in increments; each increment adds functionality Complex projects where requirements may evolve Can lose structure if increments are not well-managed
🔃 Spiral Combines waterfall + prototyping; risk assessment at each cycle High-risk, large projects requiring thorough risk mitigation Complex to manage; expensive overhead
✔️ V-Model Each development phase has a corresponding testing phase (verification + validation) Projects demanding rigorous testing at every stage Rigid; testing only done after development of each phase
🔄 Integration & Configuration (I&C) Build system using reusable components/off-the-shelf software When time is critical and reusable components exist Requires well-defined, available reusable parts
⚡ Agile Iterative, customer-focused, flexible to evolving requirements Business systems with frequently changing requirements Less suited for safety-critical or large distributed teams

Software Engineering Fundamentals

  • Systems should be developed using a managed and understood process
  • Dependability and performance are important for all system types
  • Understanding and managing software specification and requirements is critical
  • Reuse existing software whenever appropriate rather than writing from scratch

Agile Methodologies

Why Agile?

Businesses operate in rapidly changing environments where complete, stable requirements are impossible to define upfront. Agile methods focus on rapid software development and delivery, producing useful software quickly by involving customers continuously.

Agile vs. Plan-Based Development

AspectPlan-BasedAgile
Iteration occurs…Within phases/activitiesAcross phases/activities
DocumentationExtensive, formal documentsMinimal; informal communication
RequirementsDefined upfront, frozenEvolve throughout the project
Best forLarge, long-lived systems (aerospace, government)Fast-moving business systems
Team sizeLarge teams, multiple companiesSmall, co-located teams
TimelineCan take years from planning to deploymentWorking software in weeks
Customer involvementMainly at start and endContinuous throughout

Extreme Programming (XP)

XP pushes recognized good practices to the extreme. Core idea: several new versions of a system may be developed, integrated, and tested in a single day.

XP Core Practices

  • User Stories — requirements expressed as scenarios
  • Pair Programming — two programmers, one keyboard
  • Test-First Development — write tests before code
  • Continuous Integration — all tests must pass before new code is integrated
  • Small Releases — short gap between system releases
  • CRC Cards — Class-Responsibility-Collaborator cards for design

XP Process Steps

  1. Planning: Identify stories, prioritize, group into releases. Estimate velocity from iteration 2 onwards.
  2. Design: Build task cards and CRC cards
  3. Testing: Acceptance tests derived from user stories (customer tests)
  4. Release: Decide the software increment for next iteration

Scrum

Scrum is a plan-based agile framework that provides structure for organizing agile projects and gives external visibility. It does not mandate specific practices (like pair programming), making it easier to integrate with existing company policies.

Week Four — Part A

CMMI — Capability Maturity Model Integration

What is CMMI?

CMMI measures Software Process Maturity — how good an organization is at delivering high-quality software predictably. Think of it as: Process Area = a skill, Capability = how good you are at that skill, Maturity = how good your overall process is.

The 5 Levels of CMMI

LevelNameCafé AnalogySoftware Reality
1Initial (Chaotic)No recipes; some days food is great, other days terrible. Success depends on luck.Unpredictable; success depends on individual heroics, not process
2Managed (Basic)Basic recipes created; orders written down; inventory tracked; complaints monitoredBasic planning, tracking, requirements management in place. Documentation exists.
3Defined (Standardized)Every recipe, process, and routine is fully documented; staff trained on standard proceduresProcesses are well-defined, standardized across the organization
4Quantitatively ManagedTrack average prep time, defect rates, customer satisfaction scores with metricsProcesses measured and controlled using quantitative data
5OptimizingContinuously improve recipes, invent new dishes, use data to innovateFocus on continuous process improvement and innovation
Remember: Higher CMMI level = more predictable outcomes. A Level 1 organization struggles to deliver on time; a Level 5 organization continuously improves. Most organizations target Level 3.

Week Four — Part B

Introduction to Software Requirements

What are Requirements?

Requirements are descriptions of the services a system should provide and its operational constraints. The process of finding, analyzing, documenting, and checking these is called Requirements Engineering (RE).

User Requirements vs. System Requirements

TypeDefinitionAudienceExample (Sustainable Shopping App)
User Requirements High-level, abstract statements in natural language + diagrams. What the system should do for users. Customers, managers, non-technical stakeholders "The user should be able to scan a product barcode and see its sustainability rating."
System Requirements Detailed, formal technical descriptions of functions, services, and constraints. What developers must implement. Software developers, architects "SR1.1: The system shall allow users to scan EAN-13 and UPC-A barcodes using the device camera. SR1.2: Product data retrieved within 3 seconds."

System Stakeholders

A stakeholder is anyone who is affected by or has a legitimate interest in the system. Example for Mentcare System:

👤 Patients

Whose information is recorded

🩺 Doctors

Assess and treat patients using system

👩‍⚕️ Nurses

Coordinate consultations

💻 IT Staff

Install and maintain the system

⚖️ Ethics Manager

Ensures ethical guidelines are met

📊 Healthcare Managers

Obtain management reports

Functional & Non-Functional Requirements

Functional vs. Non-Functional Requirements

AspectFunctional RequirementsNon-Functional Requirements
DefinitionStatements of services the system should provide; how it reacts to inputs; what it should do (or not do)Constraints on services or functions; timing, development process, standards
ScopeApply to specific system features or servicesUsually apply to the system as a whole
Example (Medical App)"Each staff member shall be identified by an 8-digit employee number""The system must be reliable — it cannot fail while delivering insulin"
TypesFunctional user requirements & functional system requirementsProduct requirements, organizational requirements, external requirements

Non-Functional Requirements — Categories

CategoryDescriptionExample
🚀 Product RequirementsSpecify the behavior of the productSystem shall respond within 2 seconds; availability ≥ 99.9%
🏢 Organizational RequirementsDerived from policies, procedures, standards of the organizationDevelopment must follow ISO 9001 standards
⚖️ External RequirementsArise from external factors — legislation, regulations, ethicsSystem must comply with GDPR data protection laws

Software Requirements Specification (SRS)

An official document stating what system developers should implement. Based on IEEE 1998-830 standard.

When SRS is Essential

  • Development is outsourced
  • Different teams develop different components
  • Detailed requirements analysis is mandatory

When SRS May Not Be Needed

  • Software-first approach / quick prototypes
  • Agile development (user stories replace SRS)
  • Internal small-team projects

Ways to Write Requirements

MethodDescriptionWhen Used
Natural LanguageFree-form sentences; easy to write but prone to ambiguityUser requirements
Structured SpecificationTemplate-based format with defined fields (function name, inputs, outputs, side effects)System requirements
Use Cases / Graphical NotationUML diagrams (use cases, sequence diagrams)Interaction modeling
Mathematical ModelsFormal notation e.g. ResponseTime ≤ 2s; S={Idle, Monitoring, Delivering, Alarm}Safety-critical systems
Common Trap: Imprecision in requirements leads to disputes. A developer will interpret an ambiguous requirement in a way that simplifies implementation — not necessarily what the customer wanted.

Week Six — Part A

Requirements Engineering

Three Core Activities of Requirements Engineering

1. Elicitation

Discovering requirements by interacting with stakeholders (interviews, observation, workshops, scenarios)

2. Specification

Converting requirements into a standard form (SRS document)

3. Validation

Checking that requirements actually define what the customer wants

RE Process — Spiral Approach

Requirements engineering is iterative, not linear. The spiral model allows entry/exit at any stage once sufficient requirements are gathered.

Requirements Change

Requirements always change because:

  • People develop a better understanding of what they need
  • The buying organization changes
  • Hardware/software/organizational environment is modified

Changes must be managed to understand their impact on other requirements and implementation costs.

Feasibility Study

Before committing to full requirements engineering, a feasibility study checks whether: the system contributes to organizational objectives, can be built within budget/schedule constraints, and can be integrated with existing systems.

Week Six — Part B

System Modelling

What is System Modelling?

System modelling is the process of developing abstract models — simplified graphical representations of a system, using UML diagrams. Each model presents a different perspective or view of the system.

System Modelling Perspectives

PerspectiveWhat it ShowsUML Diagram Type
📦 ContextSystem boundaries — what is and isn't part of the systemContext diagram, Package diagram
🔗 InteractionHow the system interacts with users and other systemsUse Case diagrams, Sequence diagrams
🏛️ StructureOrganization of the system — components and relationshipsClass diagrams, Component diagrams
⚙️ BehaviorHow the system responds to events; state changesState machine diagrams, Activity diagrams

Where are Models Used?

PhasePurpose of Model
Requirements EngineeringClarify and discover detailed requirements; discuss existing system strengths/weaknesses
Design ProcessDescribe the system to engineers who will implement it
After ImplementationDocument system structure and operation for maintenance teams
Important: A system model is NOT a complete representation — it is an abstraction. You selectively include only what is needed for the perspective being shown.

Model-Driven Engineering (MDE)

In MDE, executable systems are automatically generated from structural and behavioral models. Tools: Eclipse Modelling Framework, Papyrus, IBM Rational Rhapsody.

Week Seven

System Models — Interaction & Behavioral Models

Recap: Structural vs. Interaction Models

Model TypePurpose
Context ModelDecide system boundaries — what is in and out of scope
Structural ModelDisplay organization of system — components and their relationships
Interaction ModelModel how the system interacts with users, other systems, and internal components
Behavioral ModelShow how the system responds to events/stimuli (state machines, activity diagrams)

Interaction Models

Three types of interaction to model:

  • User interaction — helps identify user requirements and UI design
  • System-to-system interaction — highlights communication problems at interfaces
  • Component interaction — helps check if proposed system structure delivers required performance and dependability

Use Case Modeling

Used to model interactions between a system and external agents (humans or other systems). Shows what the system does, not how.

Sequence Diagrams

Model interactions between system components, showing the sequence of messages passed between objects/components over time. Provide more detail than use cases.

Use Case vs. Sequence Diagram: Use cases show what the system does at a high level. Sequence diagrams show how components interact in detail to realize a use case.

Software Architecture

What is Architectural Design?

Architectural design is concerned with the organization and structure of a software system — identifying the main structural components and the relationships between them. Output: Architectural Model.

Architecture: In the Small vs. In the Large

ScopeDescription
Architecture in the SmallArchitecture of an individual software system — how it is decomposed into components
Architecture in the LargeArchitecture of complex enterprise systems that include multiple systems across different companies (distributed systems)

Why is Architecture Important?

  • Affects performance, robustness, distribute-ability, and maintainability
  • Individual components fulfill functional requirements, but architecture influences non-functional requirements
  • Changing architecture is expensive — must be decided early

Common Architectural Patterns

PatternKey IdeaExample
MVC (Model-View-Controller)Separates data (Model), presentation (View), and logic (Controller)Web applications; Django, Spring MVC
LayeredSystem organized into layers; each layer provides services to the layer above itOperating systems; enterprise applications (Presentation → Business Logic → Data)
RepositoryComponents share data through a central repository/databaseIDEs, CAD systems, compiler tools
Client-ServerClients request services from centralized serversEmail, web browsers, online banking
Pipe and FilterComponents (filters) process data; pass result through pipes to next componentUnix shell commands, data processing pipelines
Event-BasedComponents communicate by broadcasting and subscribing to eventsGUI frameworks, IoT systems

Week Nine

Design & Implementation — OO Design with UML

Design vs. Implementation

ConceptDescription
Architectural PatternHow the whole system (spaceship) is organized — the big picture
Design PatternHow individual components within the architecture communicate and solve common problems
ImplementationRealizing the design as actual working code; fixing and running the system

Object-Oriented System

A set of interacting objects that each maintain their own state/data and provide operations (methods) on that state to fulfill an objective. Changes to one class (object) should not affect others.

OO Design Steps

  1. Understand context — understand how the system interacts with its environment
  2. Design system architecture — high-level structure using architectural patterns
  3. Identify principal objects — what are the key objects/classes?
  4. Develop UML design models — class diagrams, sequence diagrams, state machines
  5. Specify interfaces — define how components interact (APIs, method signatures)

Design Patterns

Reusable solutions to commonly occurring design problems. Three main categories:

CategoryPurposeExamples
CreationalHow objects are createdSingleton, Factory, Builder, Prototype
StructuralHow classes/objects are composedAdapter, Decorator, Facade, Proxy
BehavioralHow objects communicate and divide responsibilityObserver, Strategy, Command, Iterator
Modular Design Principle: If Life Support stops, does Navigation stop too? No — because of modular/layered design. Each module works independently. This is why OO design and design patterns matter: they enforce loose coupling and high cohesion.

Architectural Patterns — Microservices, P2P, Event-Based

Microservices Architecture

Application consists of small, independent services that communicate over a network. Each service handles one specific function and can be developed, deployed, and scaled independently.

FeatureDescription
IndependenceEach service can use its own programming language, database, and frameworks
Own DatabaseEach microservice has its own separate database (no shared DB)
CommunicationREST APIs, gRPC, or asynchronous messaging (Apache Kafka)
ScalabilityScale specific services individually (e.g., scale Payments at month-end)
ToolsDocker (containers), Kubernetes (orchestration), Eureka (service discovery), Ribbon (load balancing)

Netflix Microservices Example

👤 User Service

User data, profiles, settings

🔐 Auth Service

Login, session, token validation

🎬 Content Service

Stores show/movie metadata

⭐ Recommendation

Personalized suggestions based on history

▶️ Playback Service

Streams video via CDN

💳 Billing Service

Handles subscriptions & payments

Peer-to-Peer (P2P) Architecture

Decentralized network where each computer (peer) has equal status — every peer can act as both client and server. No central controlling server.

FeatureClient-ServerPeer-to-Peer
ControlCentralized server controlsNo central control; fully decentralized
RolesClients request, servers provideEvery peer can request AND provide
ScalabilityServer can become bottleneckScales naturally as more peers join
FailureSingle point of failure (server)No single point of failure
ExampleWeb browsing, emailBitTorrent, Bitcoin, Skype (original)

BitTorrent P2P Example

  • Download .torrent file or magnet link (contains metadata + tracker info)
  • Tracker: lightweight server that helps a peer find other peers who have the file
  • Seeders: peers with the complete file
  • Leechers: peers who have partial file (currently downloading)
  • Download from multiple peers simultaneously for speed

Event-Based Architecture

Components communicate by broadcasting events and subscribing to events they are interested in. Components are loosely coupled — they don't need to know about each other directly.

Software Testing

Purpose of Testing

Testing does two things: (1) Demonstrate that software meets requirements (Validation Testing); (2) Discover defects — find inputs where behavior is incorrect (Defect Testing).

Dijkstra: "Testing can only show the presence of errors, not their absence."

Verification vs. Validation

ConceptQuestionDescriptionAnalogy
Verification"Are we building the product right?"Checking conformance to specification — following the right stepsFollowing the recipe correctly
Validation"Are we building the right product?"Ensuring software meets customer expectations beyond just the specTasting the food to check it's good

Three Stages of Testing

Development Testing Types

TypeDescriptionKey Point
Unit TestingTesting individual program units (functions, classes) in isolationShould reveal all possible failures a component may exhibit
Integration TestingTesting combined components/interfaces; does integration work?Focus on revealing interface defects between components
System TestingTesting the complete integrated system against requirementsEmergent system behaviors are tested here

Test Case Design Approaches

ApproachDescription
Equivalence PartitioningDivide inputs into groups (partitions) where similar behavior is expected; test one value per partition
Boundary Value AnalysisTest at the boundaries/edges of input ranges (most errors occur at boundaries)
Structural (White-Box) TestingTest cases based on code structure; aim for code coverage

Basis Path Testing

A white-box technique that identifies the number of independent paths through the program using Cyclomatic Complexity.

V(G) = E − N + 2    OR    V(G) = Number of decision nodes + 1

Where: E = edges, N = nodes, V(G) = cyclomatic complexity = number of independent paths to test.

Example: Code with 2 decision nodes (IF statements) → V(G) = 2 + 1 = 3 independent paths to test. You must design at least 3 test cases to achieve full basis path coverage.

Test-Driven Development (TDD)

Write tests before writing code. Increment only when all tests pass.

  1. Identify small increment of functionality needed
  2. Write a failing test for that functionality (deliberately — shows test works)
  3. Implement the functionality
  4. Re-run test — it should now pass
  5. Move to next increment

Benefits: Code coverage, regression testing, simplified debugging, better system documentation, cleaner code.

Static V&V: Inspections and Reviews

"Static" — no code execution needed. Can find 60–90% of defects before testing. Advantages:

  • Testing can mask/hide errors that reviews catch
  • Incomplete versions of the system can be inspected
  • Tests broader qualities: standards compliance, portability, maintainability

Release Testing & Product/Process Metrics

Release Testing

Testing a specific release intended for external use (customers or other dev teams). Goal: convince the supplier that the system is good enough to be released as a product.

FeatureDescription
Testing typeBlack-box testing — tester does not know implementation; tests derived from specifications
Also calledFunctional testing — concerned only with functionality, not implementation details
Who does it?A separate testing team, not the development team

Types of Release Testing

TypeDescriptionKey Practice
Requirements-Based TestingFor each requirement, design a set of tests to validate it. It is validation testing, not defect testing.Maintain traceability records linking tests to requirements. Multiple tests per requirement are normal.
Scenario TestingDevise realistic scenarios (stories of system use) and test the system against themScenarios involve multiple system features; good for finding interaction bugs
Performance TestingTest whether system can handle intended load; find bottlenecksStress testing: push system beyond design limits to find failure point

Product Metrics

Metrics help quantify software quality attributes like reliability, maintainability, usability, and performance. They turn subjective analysis into actionable insights.

Quality AttributeMetricFormula / Note
FunctionalityRequirements Coverage (%); Standards Compliance RateISO/IEC 25010; Interfaces Working / Total Interfaces × 100
ReliabilityMTBF, Failure Rate, System UptimeMTBF = Total operating time / Number of failures
UsabilityTime to Learn; User Error Rate; User Satisfaction ScoreUSS = (users rating 4 or 5) / total users × 100
EfficiencyResponse time, Throughput, CPU/Memory usageFrom Operating Systems knowledge
MaintainabilityCyclomatic Complexity; Fan-in / Fan-out; ModularityCC = E − N + 2; Fan-in = number of callers; Fan-out = number of called modules
PortabilityInstallation Success Rate; Adaptability across platformsSuccess Rate = Successful installs / Total attempts

Function Points (FP)

Measures the size of software based on functionality — number of inputs, outputs, and files accessed — independent of programming language. Used to estimate effort and cost.

Metrics for Analysis Models

Used during requirements/analysis phase. Examples:

  • Fan-in: Number of modules that call a particular module (high fan-in = important, reused module)
  • Fan-out: Number of modules a given module calls (high fan-out = complex module, harder to maintain)
  • Cyclomatic Complexity: Measures code complexity; helps find code that is less readable, less maintainable, and more error-prone
COCOMO (Constructive Cost Model): Uses Lines of Code (LOC) as input to predict development effort and cost. Referenced in the course as an example of how metrics (LOC) feed into estimation models.

Process Metrics vs. Product Metrics

CategoryWhat it MeasuresExamples
Product MetricsAttributes of the software product — its quality, size, complexityDefect density, LOC, cyclomatic complexity, test coverage
Process MetricsAttributes of the development process — how effectively and efficiently software is producedDefect removal efficiency, time per phase, rework rate, velocity (in Scrum)
